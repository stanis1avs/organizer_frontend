# E2E Test Suite — Organizer Frontend

End-to-end тесты запускаются через Puppeteer (headless Chromium) против собранного webpack `dist/`, раздаваемого статическим сервером на порту **3001**. Бэкенд имитируется классом `MockBackend` (Node.js HTTP + WebSocket на порту **7000**). Оркестровка — Jest 29, все тесты запускаются последовательно (`--runInBand`).

**Запуск:**
```bash
npm run test:e2e               # сборка + тесты
npm run test:e2e:skip-build    # пропустить сборку, использовать существующий dist/
```

---

## Инфраструктура

### globalSetup.js
Выполняется один раз перед всеми тестами:
1. Если `SKIP_BUILD=1` и `dist/index.html` уже существует — пропускает сборку; иначе запускает `npm run build` в `organizer_frontend/`.
2. Поднимает статический HTTP-сервер, раздающий `dist/` на порту 3001.
3. Открывает `http://localhost:3001` в браузере по умолчанию системы (для визуального контроля при отладке).

### globalTeardown.js
Закрывает статический сервер после завершения всех тестов.

### helpers/mockBackend.js
Класс `MockBackend` слушает HTTP и WebSocket на одном порту:
- **WebSocket**: при подключении сразу рассылает событие `{ event: "load", ... }` со списком сообщений и токеном. При получении `{ event: "message", ... }` от клиента — эхом отправляет это сообщение обратно, имитируя подтверждение от сервера.
- **HTTP**: роут `POST /upload` возвращает `{ url: "http://localhost:7000/files/test-file.txt", filename: "test-file.txt" }` и рассылает всем WS-клиентам событие `showFile`.
- **HTTP**: роут `POST /search` возвращает массив с одним результатом поиска, содержащим поле `message: "Test search result"`.
- Метод `closeAllConnections()` принудительно закрывает все активные WS-соединения (используется в reconnect-тестах).

### helpers/browser.js
- `launchBrowser()` — запускает Puppeteer Chromium в headless-режиме.
- `openPage(browser)` — открывает `http://localhost:3001`, ждёт до 10 секунд появления `.ws-status.ws-connected` (признак успешного WS-соединения с MockBackend).

## Тест-сьют 1: `send-message.test.js` (7 тестов)

Файл проверяет основной сценарий работы с чатом: отправку и отображение текстовых сообщений.

---

### Тест 1.1 — `chat input is visible and focusable`

**Технически:** Puppeteer ищет элемент с классом `.chat_input`. Затем вызывает `elementHandle.isIntersectingViewport()` — метод, проверяющий, что хотя бы часть элемента видна в текущем вьюпорте браузера (не скрыта CSS, не обрезана overflow).

**Продуктово:** Поле ввода сообщений находится в зоне видимости при загрузке страницы. Пользователь не должен прокручивать или искать, куда писать.

---

### Тест 1.2 — `typing and pressing Enter sends a message and clears the input`

**Технически:**
1. Запоминает начальное количество `.message` элементов в DOM.
2. Кликает в `.chat_input`, вводит текст `"Hello E2E"` посимвольно через `page.type()`.
3. Нажимает Enter.
4. Ждёт через `page.waitForFunction()` одновременного выполнения двух условий: количество `.message` выросло И среди `.message_body` есть элемент с нужным текстом. Такая комбинированная проверка нужна, чтобы не поймать пустой элемент, добавленный в DOM до рендеринга контента.
5. Проверяет, что `.chat_input.value === ""` — поле очищено после отправки.

**Продуктово:** После нажатия Enter сообщение мгновенно появляется в ленте, поле ввода очищается и готово к следующему сообщению.

---

### Тест 1.3 — `clicking the send button also sends the message`

**Технически:** Повторяет сценарий 1.2, но вместо Enter кликает кнопку `.sending` (кнопка отправки рядом с полем ввода). Та же логика `waitForFunction` для надёжного ожидания.

**Продуктово:** Пользователи, предпочитающие мышь клавиатуре, могут отправлять сообщения кнопкой — оба способа работают корректно.

---

### Тест 1.4 — `empty input does not send a message`

**Технически:** Кликает Enter с пустым полем ввода, затем ждёт 500 мс (намеренная пауза — реакции не ожидается). Проверяет, что счётчик `.message` не изменился.

**Продуктово:** Случайные нажатия Enter не создают пустые сообщения, захламляющие историю.

---

### Тест 1.5 — `each sent message has a .date element in its footer`

**Технически:** Отправляет сообщение `"Date test"`, затем через `waitForFunction` ищет элемент `.message`, у которого одновременно: `.message_body` содержит нужный текст И `.date` существует и непустой (`.trim().length > 0`).

**Продуктово:** Каждое сообщение в ленте подписано временем отправки. Пользователь всегда знает, когда было сказано.

---

### Тест 1.6 — `each sent message has action tools (pin, bin, favorite)`

**Технически:** Берёт первый `.message` из DOM через `page.$()`, затем проверяет наличие вложенных элементов `.pin`, `.bin`, `.favorite_inactive` через `elementHandle.$()`.

**Продуктово:** Каждое сообщение имеет кнопки управления: закрепить, удалить, добавить в избранное. Это базовая функциональность организации переписки.

---

### Тест 1.7 — `pre-existing messages from server are rendered on page load`

**Технически:** Поднимает отдельный `MockBackend` на порту 7001 (чтобы не мешать основному на 7000). Через `setLoadData()` задаёт список начальных сообщений. Подключает Node.js WebSocket-клиент, отправляет `{ event: "load" }` и ждёт, что MockBackend ответит событием `load` — это подтверждает, что механизм начальной загрузки данных работает корректно на уровне протокола.

**Продуктово:** При открытии приложения пользователь сразу видит свою историю переписки — сообщения не исчезают между сессиями.

## Тест-сьют 2: `search.test.js` (7 тестов)

Проверяет работу AI-поиска — боковой панели, в которой пользователь вводит запрос и получает результаты от сервера.

---

### Тест 2.1 — `AI toggle button is present`

**Технически:** `page.$('.ai-toggle')` — простая проверка наличия кнопки-переключателя в DOM.

**Продуктово:** Кнопка AI-поиска всегда видна в интерфейсе и доступна пользователю.

---

### Тест 2.2 — `clicking AI toggle opens the left aside panel`

**Технически:** Кликает `.ai-toggle`, затем ждёт появления `.left-aside.left-aside-active` (таймаут 5 сек). Оба класса должны присутствовать одновременно: `left-aside` — базовый класс панели, `left-aside-active` — класс, добавляемый при открытии.

**Продуктово:** Нажатие на кнопку открывает боковую панель AI-поиска. Пользователь получает визуальный отклик.

---

### Тест 2.3 — `AI input and send button are visible in the panel`

**Технически:** Проверяет наличие `.ai_input` и `.ai_send_btn` в DOM после открытия панели.

**Продуктово:** Внутри открытой панели есть поле для ввода запроса и кнопка отправки — всё необходимое для взаимодействия с AI.

---

### Тест 2.4 — `typing a query and clicking send shows results`

**Технически:**
1. Убеждается, что панель открыта (если нет — открывает).
2. Ждёт видимости `.ai_input` через `waitForSelector` с флагом `{ visible: true }`.
3. Вводит запрос `"test query"`.
4. **Критический момент**: кнопка `.ai_send_btn` находится внутри прокручиваемой панели `.left-aside`. Puppeteer's `page.click()` использует координаты из `getBoundingClientRect()` и не может надёжно попасть в элемент, перекрытый overflow-контейнером. Поэтому используется `page.evaluate(() => document.querySelector('.ai_send_btn')?.click())` — прямой вызов DOM `.click()` обходит это ограничение.
5. Ждёт появления одного из трёх состояний: `.ai-result-item` (есть результаты), `.ai-result-empty` (ничего не найдено), `.ai-error` (ошибка).
6. Проверяет, что результатов больше нуля.

**Продуктово:** После ввода запроса и нажатия кнопки пользователь получает список результатов поиска из своей истории сообщений.

---

### Тест 2.5 — `result item contains the expected text from mock backend`

**Технически:** Читает `textContent` первого `.ai-result-item` через `page.$eval()`. Проверяет вхождение строки `"Test search result"`. MockBackend возвращает `{ message: "Test search result", ... }` — поле `message` используется компонентом `renderAIResults` в качестве заголовка результата.

**Продуктово:** Результаты поиска содержат реальный текст из сообщений, а не технические идентификаторы или пустые карточки.

---

### Тест 2.6 — `pressing Enter in AI input also triggers search`

**Технически:** Выделяет весь текст в поле тройным кликом (`clickCount: 3`), вводит новый запрос, нажимает Enter через `page.keyboard.press('Enter')`. Ждёт результатов аналогично тесту 2.4.

**Продуктово:** Пользователь может отправлять запрос нажатием Enter — не только кнопкой. Привычное поведение для поисковых полей.

---

### Тест 2.7 — `clicking AI toggle again closes the panel`

**Технически:** Кликает `.ai-toggle` повторно. Ждёт через `waitForFunction`, пока `.left-aside.left-aside-active` исчезнет из DOM (querySelector вернёт null).

**Продуктово:** Повторное нажатие на кнопку AI сворачивает панель. Пользователь может скрыть панель и вернуться к основному чату.

## Тест-сьют 3: `file-upload.test.js` (4 теста)

Проверяет загрузку файлов через вложение в чат.

---

### Тест 3.1 — `clip button is present in the chat footer`

**Технически:** `page.$('.clip')` — проверка наличия кнопки скрепки в DOM.

**Продуктово:** Кнопка прикрепления файлов всегда доступна в нижней панели чата.

---

### Тест 3.2 — `uploading a file via clip adds a file message to the chat`

**Технически:**
1. Создаёт временный файл `test-file.txt` в `os.tmpdir()` через `fs.writeFileSync`.
2. Запоминает начальное количество сообщений.
3. Кликает `.clip` → открывается скрытое меню `.clip_body`; ждёт `.clip_body.clip_active`.
4. Кликает `.clip_file` — класс `FileLoader` заменяет `.chat_input` на `<input type="file" class="input_file">` в DOM.
5. Ждёт появления `input.input_file`.
6. Вызывает `fileInputHandle.uploadFile(tmpFilePath)` — Puppeteer-метод, устанавливающий файлы на `<input type="file">` напрямую, минуя системный диалог. Это триггерит событие `change`, которое `FileLoader` обрабатывает: отправляет XHR `POST /upload` на MockBackend.
7. MockBackend возвращает `{ url, filename }` и рассылает всем WS-клиентам событие `showFile`.
8. Фронтенд получает WS-событие и добавляет сообщение с файлом в ленту.
9. Ждёт через `waitForFunction`, что счётчик `.message` увеличился.

**Продуктово:** Пользователь нажимает скрепку, выбирает файл — файл загружается на сервер, и в ленте чата появляется сообщение со ссылкой на файл.

---

### Тест 3.3 — `uploaded file message contains a file link or name`

**Технически:** Берёт последнее `.message_body` в DOM через `page.$$eval()`, проверяет регулярным выражением `/test-file\.txt/i` — имя файла должно присутствовать в тексте.

**Продуктово:** Сообщение о файле содержит понятное имя файла, а не технический URL или пустой блок.

---

### Тест 3.4 — `uploaded file message has the standard action tools`

**Технически:** Берёт последнее `.message` из DOM, проверяет наличие `.pin` и `.bin` внутри него.

**Продуктово:** Сообщения с файлами, как и текстовые, можно закрепить или удалить — единообразный UX для всех типов контента.

## Тест-сьют 4: `media-record.test.js` (4 теста)

Проверяет запись аудио и видео через браузерный API MediaRecorder. Поскольку в headless-среде нет реального микрофона/камеры, каждый тест открывает свою страницу с предварительно внедрёнными фиктивными реализациями.

**Инъекция моков (функция `openPageWithMediaMock`):**
Перед загрузкой любых скриптов страницы через `page.evaluateOnNewDocument()` в `window` внедряется:
- `FakeMediaRecorder` — имитирует `MediaRecorder`: метод `start()` переводит в состояние `recording` и через 100 мс вызывает `ondataavailable` с фиктивным `Blob`; метод `stop()` вызывает `onstop`. `isTypeSupported()` всегда возвращает `true`.
- `navigator.mediaDevices.getUserMedia` — возвращает `Promise.resolve(fakeStream)`, где `fakeStream.getTracks()` возвращает одну дорожку с методом `stop()`.

Это позволяет тестировать взаимодействие UI с классом `MediaLoader` без реального железа.

---

### Тест 4.1 — `clip_audio button is accessible after opening the clip menu`

**Технически:** Открывает страницу с медиа-моками, кликает `.clip`, ждёт `.clip_body.clip_active` (меню вложений открылось), затем проверяет наличие `.clip_audio`.

**Продуктово:** Кнопка записи аудио находится внутри меню вложений и доступна пользователю после его открытия. Кнопка не отображается по умолчанию, чтобы не перегружать интерфейс.

---

### Тест 4.2 — `clicking audio starts recording and clicking again sends a message`

**Технически:**
1. Запоминает начальное количество `.message`.
2. Открывает меню, кликает `.clip_audio` — `MediaLoader` вызывает `getUserMedia()` (возвращает `fakeStream`) и создаёт `FakeMediaRecorder`, запускает `start()`.
3. Ждёт 300 мс (асинхронная инициализация записи).
4. Ищет `.audio_icon` или `.clip_audio.recording` — элемент, появляющийся во время записи. Если найден — кликает для остановки; иначе через `page.evaluate()` ищет любой доступный элемент управления.
5. Ждёт через `waitForFunction` увеличения счётчика сообщений. Ошибка ожидания перехватывается через `.catch()` — тест помечен как некритичный (фиктивный Blob может быть слишком мал для обработки).
6. Финальная проверка: страница отвечает, `.chat_input` или `input.input_file` существует.

**Продуктово:** Пользователь может записать голосовое сообщение. Нажатие кнопки дважды — начать и остановить запись. После остановки сообщение отправляется в чат.

---

### Тест 4.3 — `clip_video button is accessible after opening the clip menu`

**Технически:** Аналогично тесту 4.1, но проверяет `.clip_video`.

**Продуктово:** Кнопка записи видео доступна в меню вложений.

---

### Тест 4.4 — `video record button can be clicked without error`

**Технически:** Открывает меню, кликает `.clip_video`, ждёт 200 мс, проверяет `page.title()` — страница не упала (не стала пустой или `about:blank`). Это smoke-тест.

**Продуктово:** Нажатие кнопки видеозаписи не приводит к крашу интерфейса. Минимальный guard-тест стабильности.

## Тест-сьют 5: `reconnect.test.js` (5 тестов)

Проверяет устойчивость WS-соединения: что приложение корректно восстанавливается после обрыва связи с сервером.

---

### Тест 5.1 — `page shows ws-connected on initial load`

**Технически:** Проверяет наличие `.ws-status.ws-connected` в DOM. Этот же элемент ожидается в `openPage()` перед стартом любого теста, так что здесь это скорее sanity check.

**Продуктово:** Индикатор состояния соединения отображается сразу после загрузки. Пользователь видит, что приложение подключено к серверу.

---

### Тест 5.2 — `dropping all connections triggers ws-reconnecting state`

**Технически:**
```js
const reconnectingPromise = page.waitForSelector('.ws-status.ws-reconnecting', { timeout: 10000 });
backend.closeAllConnections();
await reconnectingPromise;
```
**Критический порядок**: `waitForSelector` регистрируется **до** вызова `closeAllConnections()`. Состояние `.ws-reconnecting` длится около 1000 мс (первая задержка переподключения в `Request.js` — `Math.min(1000 * 2^attempts, 30000)`). Если зарегистрировать ожидание после закрытия соединений, браузер успевает переключиться обратно в `ws-connected` до того, как Puppeteer начнёт следить.

**Продуктово:** При обрыве соединения приложение немедленно показывает пользователю индикатор "переподключение..." — честная обратная связь вместо зависания.

---

### Тест 5.3 — `client reconnects and returns to ws-connected`

**Технически:** Ждёт `.ws-status.ws-connected` с таймаутом 15 сек (с учётом задержки переподключения ~1 сек и времени TLS/HTTP handshake). MockBackend продолжает слушать порт 7000 — новое WS-соединение успешно устанавливается.

**Продуктово:** После кратковременного обрыва приложение автоматически восстанавливает соединение. Пользователю не нужно перезагружать страницу.

---

### Тест 5.4 — `messages sent after reconnect are delivered`

**Технически:** После восстановления соединения вводит и отправляет `"Post-reconnect message"`. Ждёт через ту же `waitForFunction`-логику, что и в `send-message.test.js`: одновременная проверка счётчика и наличия текста. Затем верифицирует через `$$eval`.

**Продуктово:** После переподключения чат полностью работоспособен — сообщения доставляются, история не теряется. Пользователь может продолжить общение без ручного перезапуска.

---

### Тест 5.5 — `second disconnect/reconnect cycle also recovers`

**Технически:** Повторяет сценарий тестов 5.2 + 5.3 в рамках одного теста: предварительная регистрация `waitForSelector`, затем `closeAllConnections()`, затем ожидание `ws-connected`.

**Продуктово:** Приложение выдерживает несколько последовательных обрывов соединения за сессию. Например, при нестабильном интернете пользователь не потеряет работоспособность чата.
